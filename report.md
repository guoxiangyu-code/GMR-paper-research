# Soccer-GMR 系统性错误分析报告

> 模型：Moment-DETR-GMR | 数据集：Soccer-GMR Standard Split | 日期：2026-06-23

---

## 1 研究目标

对 Moment-DETR-GMR 基线模型在 Soccer-GMR 基准上进行系统性、数据驱动的错误分析，量化失败模式，通过反事实 Oracle 修复对瓶颈排序，并实现针对最高影响力瓶颈的最小改进模块。所有结论均基于官方评估脚本的可复现统计量。

---

## 2 实验设置

### 2.1 模型架构

| 组件 | 描述 |
|------|------|
| 骨干 | DETR-style Transformer encoder-decoder（2 层, 8 头） |
| 视频特征 | CLIP (512d) + SlowFast (2304d) + TEF (2d) = 2818d → 256d |
| 文本特征 | CLIP (512d) → 256d |
| 预测头 | class_embed (2类) + span_embed (center,width) + saliency_proj + exist_head |
| GMR Adapter | max-pool 10 decoder queries → 2 层 MLP → existence logit |
| 查询槽数 | 10 个可学习 query embedding |

| Test | 1,036 | 544 (384 single + 160 multi) | 492 | 160 |

## 3 Phase 1：逐查询错误分析详情表

### 3.1 方法

创建 `eval/error_analysis.py`，复用 `eval/metrics.py` 中的 `greedy_match` 函数，为每个查询生成：

- **场景标签**：`null_set`（|G|=0）、`single`（|G|=1）、`multi`（|G|≥2）
- **匹配统计**：在 IoU 阈值 {0.5, 0.55, ..., 0.95} 下的匹配/遗漏/额外计数
- **强制匹配 IoU**：无 IoU 阈值限制的 greedy match IoU 值
- **存在性分数**：`pred_exist_score` 及来源
- **错误类型标签**：`rejection_FP`、`rejection_FN`、`single_miss`、`single_hit`、`multi_partial_miss` 等

### 3.2 错误类型分布（Test, 1036 queries）

| 错误类型 | 数量 | 占总量比 |
|----------|------|----------|
| correct_reject | 321 | 31.0% |
| single_miss | 223 | 21.5% |
| rejection_FP | 171 | 16.5% |
| single_hit_plus_extra | 161 | 15.5% |
| multi_partial_miss | 79 | 7.6% |
| multi_total_miss | 64 | 6.2% |
| multi_hit_plus_extra | 17 | 1.6% |

## 4 Phase 2：六大失败模式量化

### 4.1 Rejection FP（空集被错误接受）

- **171/492** 空集查询被错误接受（34.8%）
- 空集分数高度集中在阈值附近（mean=0.394, std=0.041），微小阈值变化即产生大量影响
- 分数分布：min=0.304, max=0.594，全部落在 [0.3, 0.6] 的窄区间

### 4.2 Rejection FN（正例被错误拒绝）

- **190/544** 正例查询被错误拒绝（34.9%）
- 102 个单时刻 + 88 个多时刻正例丢失
- 存在性头校准严重不足：正例均值 0.638 vs 空集均值 0.394，分布重叠严重

### 4.3 Multi-moment Miss（多时刻遗漏）

- 160 个多时刻查询共 366 个 GT 时刻，仅匹配 122 个（33.3% recall @IoU≥0.5）
- 43.8% 为"仅命中首个时刻"
- 首个时刻命中率仅 8.1%——模型在最容易的时刻上表现也很差

### 4.4 Multiple Detections（冗余检测）

- 100% 正例查询有额外预测（模型总是输出 10 个窗口）
- 这是架构设计导致的默认行为，评估已通过 greedy matching 处理
- **不属于真正的失败模式**

### 4.5 Inaccurate Boundaries（边界不精确）

- 84.1% 的强制匹配 IoU < 0.3，仅 9.3% 达到 IoU ≥ 0.5
- 49 对（6.5%）为"接近命中"（0.3 ≤ IoU < 0.5）
- **系统性时间偏移**：低 IoU 对（<0.1）的平均起始偏移 = -25.53s，结束偏移 = -24.96s
- 模型检索到视频中错误的时间段，而非边界微小偏差

### 4.6 边界偏移详细分析

| IoU 范围 | 匹配对数 | 平均起始偏移(s) | 平均结束偏移(s) | 平均预测时长(s) | 平均 GT 时长(s) |
|-----------|----------|-----------------|-----------------|-----------------|-----------------|
| [0, 0.1) | 585 | -25.53 | -24.96 | 7.74 | 7.16 |
| [0.1, 0.3) | 46 | -2.49 | -1.73 | 7.48 | 6.72 |
| [0.3, 0.5) | 49 | -0.50 | +0.62 | 8.04 | 6.92 |
| [0.5, 1.0) | 64 | -0.12 | +0.50 | 7.69 | 7.06 |

> 低 IoU 预测平均偏移 ~20-25 秒，属于**系统性时间错位**，非边界微调可解决。

### 4.7 存在性分数分布重叠

| 集合 | 均值 | 标准差 | 最小值 | 最大值 |
|------|------|--------|--------|--------|
| 空集 (492) | 0.3941 | 0.0410 | 0.3044 | 0.5940 |
| 正例 (544) | 0.6377 | 0.2957 | 0.3127 | 1.0000 |

空集分数高度集中（std=0.041），正例分数分散（std=0.296），两者在 [0.3, 0.6] 区间严重重叠。

## 5 Phase 3：反事实 Oracle 修复排序

### 5.1 方法

对预测文件施加 Oracle 修复，每次仅修复一种失败模式，重新运行官方评估，衡量各修复的独立指标增益。

### 5.2 Oracle 修复结果（Test Set）

| 修复方案 | AUROC | Rej-F1@0.4 | G-mIoU@1 | mAP | mR@5 | mR+@5 | mIoU@1 |
|----------|-------|------------|----------|-----|------|-------|--------|
| Baseline | 72.09 | 64.01 | 35.84 | 7.52 | 12.96 | 0.84 | 12.42 |
| fix_boundaries | +0 | +0 | **+25.45** | **+92.48** | **+87.04** | **+99.16** | **+87.58** |
| fix_false_positives | **+27.91** | **+19.81** | +16.51 | +0 | +0 | +0 | +0 |
| fix_multi_moment | +0 | +0 | +0 | +0 | +0 | +0 | +0 |
| fix_multiple_detections | +0 | +0 | +0 | -3.67 | -8.74 | -0.75 | +0 |

### 5.3 瓶颈排序

| 排名 | 瓶颈 | 总指标增益 | 可行性 |
|------|------|-----------|--------|
| 1 | 边界不精确 | 391.71 | **不可行** — 系统性 ~20s 时间错位，需模型重训练 |
| 2 | False Positive | 64.23 | **可行** — 存在性分数校准可后处理实现 |
| 3 | 冗余检测 | 13.16 | 有害 — 移除"额外"预测降低 recall |
| 4 | 多时刻遗漏 | 0.00 | 当前模型太弱，非多时刻专属问题 |

### 5.4 关键洞察

1. **边界修复理论上限极高**（mAP +92.48），但模型存在系统性时间错位（~20s），无法通过后处理修复
2. **FP 修复最具可操作性** — 存在性分数校准可直接减少 rejection_FP
3. **多时刻修复增益为零** — 因为模型连第一个 GT 都无法在 IoU≥0.5 匹配
4. **冗余检测移除有害** — "额外"预测在低 IoU 阈值下可能匹配 GT

## 6 Phase 4：存在性分数校准模块

### 6.1 方法：温度缩放（Platt Scaling）

对原始存在性分数施加温度缩放：

```
calibrated_score = sigmoid(logit(score) / T)
```

- **T < 1**：锐化 sigmoid，将中间分数推向 0 或 1
- **T > 1**：软化 sigmoid，将分数推向 0.5

在验证集上以 max_rej_f1 为目标搜索最优温度 T 和阈值。

### 6.2 最优参数

| 参数 | 值 | 说明 |
|------|----|------|
| Temperature (T) | 0.30 | 在 val 上搜索 [0.3, 5.0] 范围 |
| Threshold | 0.63 | 在 val 上搜索 [0.3, 0.8] 范围 |
| 优化目标 | max_rej_f1 | 最大化拒绝类 F1 |

### 6.3 核心代码

模块位于 `models/moment_detr_gmr/exist_calibrator.py`，提供：

- `ExistenceCalibrator.fit(val_sub, val_gt)` — 在验证集上学习参数
- `ExistenceCalibrator.apply_to_predictions(submission)` — 对预测施加校准
- `ExistenceCalibrator.save()/load()` — 参数持久化

### 6.4 测试集结果

| 指标 | Baseline | Calibrated | 增益 |
|------|----------|------------|------|
| **G-mIoU@1** | 35.84 | **49.64** | **+13.80** |
| **G-mIoU@3** | 32.89 | **46.82** | +13.93 |
| **G-mIoU@5** | 32.14 | **44.34** | +12.20 |
| **Rej-F1@0.4** | 64.01 | **74.19** | **+10.18** |
| **Acc@0.4** | 65.15 | **68.44** | +3.29 |
| AUROC | 72.09 | 72.09 | +0.00 |
| mAP | 7.52 | 7.52 | +0.00 |
| mR@5 | 12.96 | 12.96 | +0.00 |
| mR+@5 | 0.84 | 0.84 | +0.00 |
| mIoU@1 | 12.42 | 12.42 | +0.00 |

> AUROC 不变是预期的：温度缩放是单调变换，不改变排序。

### 6.5 验证集结果

| 指标 | Baseline | Calibrated | 增益 |
|------|----------|------------|------|
| G-mIoU@1 | 35.41 | **46.81** | +11.40 |
| Rej-F1@0.4 | 64.76 | **70.26** | +5.50 |

### 6.6 消融实验

| 组件 | G-mIoU@1 | Rej-F1@0.4 | 说明 |
|------|----------|-------------|------|
| Baseline | 35.84 | 64.01 | — |
| + 仅温度缩放 (thd=0.4) | **49.64** | **74.19** | 温度缩放重塑分数分布 |
| + 仅阈值调整 (T=1.0, thd=0.6) | 35.84 | 64.01 | 阈值调整对 G-mIoU 门控无效 |
| + 完整校准 (T=0.30, thd=0.63) | **49.64** | **74.19** | 与仅温度缩放等价 |
| + 硬门控 (无分数缩放) | **49.64** | **74.19** | 与软门控效果相同 |

**结论**：温度缩放是关键组件。`eval_main.py` 使用自己的 G-mIoU 门控阈值，直接修改预测中的阈值不改变 G-mIoU 门控行为；但温度缩放重塑了分数分布本身，使更多空集分数低于 G-mIoU 门控阈值。

### 6.7 交叉验证：错误类别特异性

| 错误类型 | Baseline | Calibrated | 变化 |
|----------|----------|------------|------|
| **rejection_FP** | **171** | **22** | **-149 (-87%)** |
| correct_reject | 321 | 470 | +149 |
| single_miss | 223 | 223 | 0 |
| multi_partial_miss | 79 | 79 | 0 |
| multi_total_miss | 64 | 64 | 0 |
| single_hit_plus_extra | 161 | 161 | 0 |
| multi_hit_plus_extra | 17 | 17 | 0 |

> 模块**仅减少目标错误类别**（rejection_FP），不影响其他类别。

## 7 结论与展望

### 7.1 核心发现

1. **存在性分数校准是最具可操作性的改进方向**：温度缩放 (T=0.30) 将 G-mIoU@1 从 35.84 提升至 49.64 (+38.6%)，Rej-F1@0.4 从 64.01 提升至 74.19 (+15.9%)

2. **边界不精确是理论最大瓶颈但不可行**：Oracle 修复可带来 mAP +92.48，但模型存在 ~20s 系统性时间错位，需重新训练模型

3. **本地 checkpoint 质量远低于论文**：mAP 7.52 vs 18.69，定位能力极弱（84% 匹配 IoU < 0.3），建议获取官方 checkpoint 验证

4. **存在性头校准严重不足**：空集与正例分数分布严重重叠，模型对"该视频是否存在相关时刻"的判断接近随机

### 7.2 后续工作建议

| 优先级 | 方向 | 预期收益 | 方法 |
|--------|------|----------|------|
| P0 | 获取官方 checkpoint | 验证结论泛化性 | 联系论文作者 |
| P1 | 存在性头训练改进 | AUROC 提升 | Focal loss + 类别平衡采样 |
| P2 | 时间定位改进 | mAP 大幅提升 | 增加训练 epoch / 改进 span loss |
| P3 | 多时刻检索 | mR+@5 提升 | Coverage-aware loss + query 去重 |
