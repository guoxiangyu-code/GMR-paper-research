## 1. 基线模型特性 (Phase 1 发现)

### 1.1 pred_exist_score 分布异常
- **发现日期**: 2026-06-24
- **详情**: 所有1036个测试样本的 pred_exist_score 均大于默认阈值 0.4
  - 默认阈值下: TP=544, FP=492, TN=0, FN=0 → Rej-F1=0.00
  - 阈值0.5: TP=409, TN=197, FP=295, FN=135 → Rej-F1=47.82
  - 阈值0.55: TP=306, TN=369, FP=123, FN=238 → Rej-F1=67.15
  - 阈值0.6: TP=244, TN=455, FP=37, FN=300 → Rej-F1=72.98
- **影响**: G-mIoU@1 从4.49 (θ=0.4) → 23.01 (θ=0.5) → 39.31 (θ=0.55)
- **结论**: exist_head 的分数校准 (calibration) 存在问题, 分数整体偏高

### 1.2 多时刻检索严重不足
- **发现日期**: 2026-06-24
- **详情**: 
  - mR+@1 = 0.0 (在所有 IoU 阈值下均为0)
  - mR+@3 = 0.19 (仅在 IoU 0.5~0.6 下有微弱命中: 0.62%)
  - mR+@5 = 0.97 (IoU 0.5~0.6: 2.92%, IoU 0.65~0.75: 0.31%)
- **影响**: 160 个多时刻 query 几乎全部只能命中第一个 moment (或一个都没命中)
- **推测**: Moment-DETR 的 decoder queries 可能都收敛到相同的时刻区域

### 1.3 定位精度 (mAP) 分析
- **发现日期**: 2026-06-24
- **详情**:
  - mAP=8.09, 高于论文 Moment-DETR-GMR (7.52) 但远低于 FlashVTG-GMR (24.62)
  - mR@1=3.1 (仅 6.41% 在 IoU≥0.5 时 Top-1 命中)
  - mIoU@1=10.37, 说明 Top-1 预测与 GT 的平均 IoU 仅约 0.1
- **结论**: 定位质量本身就不高, 但这可能是特征瓶颈 (CLIP+SlowFast vs 端到端)

### 1.4 数据集结构
- **发现日期**: 2026-06-24
- **详情**:
  - 训练集: 5185 (正:2722, 负:2463), 正负比约 1.1:1
  - 测试集: 1036 (正:544, 负:492), 正负比约 1.1:1
  - 正样本中: single-moment 占 ~71% (384/544), multi-moment 占 ~29% (160/544)
  - 多时刻样本可能有 2~N 个 GT moments (需进一步统计分布)

## 2. Error Analysis 发现 (Phase 2 发现)

### 2.1 拒识误报 (FP) 分析
- **发现**：大量负样本的预测存在性得分偏高。由于模型未能有效拒识 null-set，导致它们被作为正样本计入惩罚。

### 2.2 拒识漏报 (FN) 分析  
- **发现**：相对于 FP，漏报的影响相对较小。在最佳的 F1 阈值下，主要瓶颈依旧是 FP。

### 2.3 多时刻漏检分析
- **发现**：绝大多数多时刻 query 只能命中第一个时刻。但直接补齐漏掉的 GT 时刻，实际上无法立刻提升 G-mIoU，因为模型同时输出了大量多余框（分母变大）。

### 2.4 多检 (Over-detection) 分析
- **发现**：不论 GT 数量，模型几乎总是输出 10 个预测框，导致严重的过检。未命中的多余框大幅增加了 set-level IoU 的分母。
- **结论**：过检是拖累 G-mIoU@1 的最直接因素。

### 2.5 边界不准分析
- **发现**：已命中框的边界存在整体后移的系统性偏移 (Δstart=+7.4s, Δend=+8.1s)。
- **结论**：这是导致时序定位 mAP 低下的主要原因。

## 3. Oracle 修复发现 (Phase 3 发现)

### 3.1 核心结论排序
通过逐一进行反事实（Oracle）修复（引入 G-mIoU@3 作为辅助观察口径防止截断假象），我们量化了各类错误对指标的绝对影响：

1. **Fix Over-detection (修复过检)** 🏆：
   - 移除未匹配的预测框后，G-mIoU@1 猛增 **+18.92**，而 G-mIoU@3 增益更高达 **+21.31**。
   - 这证明了“过检”是由于模型在后续预测中输出了大量低质量废框，严重摊薄了 set-level IoU 的分母。这是单点修复收益最大的方向。
2. **Fix FP (修复误报)**：
   - 完美拒识负样本后，G-mIoU@1 增加 **+11.87** (达 51.18)。Rej-F1 增加 **+13.37**。
3. **Fix FN (修复漏报)**：
   - 强制将因分数过低而被错拒的正样本拉回后，G-mIoU@1 仅提升 **+0.80**，但 Rej-F1 大幅回升 **+18.56**。
   - 说明分类缺口确实存在于 FN 中，但由于这些样本内部的定位质量（过检等）依然很差，强行放进来也无法拯救 G-mIoU。
4. **Fix Boundary (修复边界)**：
   - 使用真实的 GT 边界替换匹配框后，**mAP 暴涨 +30.74** (达 38.83)，但 G-mIoU@1 仅提升 +5.12。
5. **Fix Multi-miss (修复多时刻漏检)**：
   - 即使在 G-mIoU@3 或 mR+@5 口径下，补齐漏掉的 GT 时刻依然是 **零增益**。多时刻漏检本身并不是主要瓶颈，核心问题在于预测集充满了噪声。

### 3.2 方案设计指引
根据 Oracle 排序，我们的系统性改进方向被明确锁定为：**存在性校准和多余预测抑制 (后处理方向)**，也就是接下来的 Phase 4。

## 4. 关键参考

### 评测命令
```bash
python eval/eval_main.py \
  --submission_path <pred> \
  --gt_path data/label/Standard/test.jsonl \
  --save_path <out>
```

### 可选参数
- `--gmiou_cls_threshold`: G-mIoU 的分类阈值 (默认 0.4)
- `--iou_thds`: IoU 阈值列表 (默认 [0.5, 0.55, ..., 0.95])
- `--topK_list`: Top-K 列表 (默认 [1, 3, 5])
- `--match_number`: 每个 query 最多评估的预测数 (默认 10)
