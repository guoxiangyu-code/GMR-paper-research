这套"val 只留正样本"的安排,和论文第 3.3 节的评测协议是严丝合缝的。论文把指标分成三组:null-set 拒识(在全部查询上算)、时序定位(只在正样本 Q⁺ 上算)、以及端到端 G-mIoU(在全部查询上算)。论文在 3.3.2 明确写道,定位指标"仅在正样本查询 Q⁺ 上评估,使定位分数纯粹反映时序定位能力,不受各模型拒识特性差异的影响"。 arXiv
而 val 这一支在代码里走的正是"定位导向"的轻量评测:eval_epoch_post_processing 中只有当 split=="val" 时才会调用 eval_submission(来自 standalone_eval,即 Moment-DETR 沿用下来的传统评估器),用于训练期的 loss 监控与模型选择(挑 best.ckpt)。这个评估器算的是 mAP、R@k 这类定位指标,而它们在数学上只对"有真值时刻"的查询有定义——null-set 查询没有任何 GT 窗口,recall/IoU 要么无定义要么恒为 0,放进来只会污染模型选择的信号。因此 val 把负样本过滤掉,是为了让验证期的定位指标干净、可用。
真正需要负样本的拒识与端到端指标,则交给另一套独立的官方工具 eval/eval_main.py 在 test 上完成。那里用 normalize_ground_truth(..., drop_empty_gt=False) 显式保留空 GT 样本,compute_gmr_cls 也必须同时拿到正样本(y_true=1)和负样本(y_true=0)才能算出 AUROC 和 Rej-F1。这就形成了清晰的分工:val = 训练期、正样本、定位导向、用于选模型;test = 全量含负样本、走官方工具、评估完整 GMR(含拒识)。​ GitHub
一个对你做 error analysis 的实际提醒
这一点对你前面讨论的"先做错误分析再定位瓶颈"很关键:由于 val 推理在数据集层面就过滤了负样本,你无法直接用现成的 val 预测去分析 null-set 误报(submission 里根本没有负样本查询)。如果你想在验证集上诊断拒识错误,需要做两件事之一——要么在生成 val 预测时把 keep_empty_gt 强制设为 True(让 null-set 样本进入推理),要么直接把含负样本的 val 标签文件喂给官方 eval/eval_main.py 并配上包含负样本预测的 submission。换句话说,要复现论文 Table 2 那种带 Rej-F1/AUROC 的完整结果,基准入口应当是 test 这条 keep_empty_gt=True 的路径,而不是默认的 val 路径。
