### **病因已确诊为 query 引力坍缩,下一步该上的不是更强的隐式加噪,而是一个"只在过度重叠时才发力、且不伤害匹配 query"的时序排斥损失——下面给出可直接接进 `SetCriterion` 的实现与四步落地规划**

你这张七列对比表把故事彻底跑通了:#2(+SA+QD)和 #3(-SA+QD)从两个方向证明了**数据侧隐式加噪打不过 Soccer-GMR 主事件的引力**——#3 删了去重器全塌、#2 留了去重器但新 query 照样向主事件坍缩还把 mAP 拖崩。GT 覆盖率 0.55 是冒烟枪。结论无可辩驳:**必须从 loss 层施加显式排斥**。我先把损失写出来,再讲怎么稳妥地接。

**核心设计原则(决定成败的三条,先讲清再看代码)**

第一,**绝不能裸排斥所有 query 对**。Soccer-GMR 里真实的多个 moment 时间上常常挨得近甚至轻微重叠,如果无脑惩罚所有重叠,会把"本该覆盖相邻真实 moment"的 query 也推开,反而降低覆盖率。所以排斥必须带 margin——**只惩罚"重叠超过容忍阈值"的部分**,适度重叠不罚。

第二,**匹配上 GT 的 query 要保护**。被 Hungarian 匹配上的 query 是各 GT 的"正主",它们之间的距离应该由 GT 本身决定,不该被排斥项乱推。所以排斥项**主要作用在"未匹配的空闲 query"身上**——正是这些空闲 query 现在全堆到主事件上,把它们赶开,逼它们要么去覆盖次要 moment、要么退到背景(被拒答头判 NONE)。

第三,**这是 Idea 1 之上的增量,守护底盘不能动**。loss 权重要小、要可配,先小步加,盯着 mAP / Rej-F1 不破。

**实现:`temporal_diversity_loss`(放进 `models/moment_detr_gmr/diversity_loss.py`)**

````python
# -*- coding: utf-8 -*-
"""阶段三:时序多样性/排斥损失,治 Soccer-GMR 的 query 引力坍缩。
作用:对"未匹配空闲 query"之间(及空闲 vs 匹配)预测 span 的过度时序重叠施加惩罚,
逼空闲 query 离开主事件——要么去覆盖次要 moment,要么退背景被拒答。
保护:被 Hungarian 匹配上的 query 之间不互相排斥(它们的间距由 GT 决定)。
"""
import torch
from models.moment_detr_gmr.utils.span_utils import (
    generalized_temporal_iou, span_cxw_to_xx)


def temporal_diversity_loss(pred_spans, matched_src_idx, iou_margin=0.5,
                            protect_matched=True, use_giou=False, eps=1e-6):
    """
    pred_spans:      [B, N, 2] (cx, w) 归一化预测 span(用 outputs["pred_spans"])
    matched_src_idx: List[Tensor],每个 batch 样本被 Hungarian 匹配的 query 下标
    iou_margin:      容忍阈值;tIoU 超过此值的部分才被惩罚(适度重叠不罚)
    protect_matched: True 时不惩罚 matched-matched 对
    返回标量 loss。
    """
    B, N, _ = pred_spans.shape
    device = pred_spans.device
    total = pred_spans.new_zeros(())
    n_pairs = 0
    for b in range(B):
        spans_xx = span_cxw_to_xx(pred_spans[b])           # [N,2] (st,ed)
        if use_giou:
            iou = generalized_temporal_iou(spans_xx, spans_xx)  # [N,N], 含负值
            iou = iou.clamp(min=0.0)
        else:
            iou = _pairwise_tiou(spans_xx)                  # [N,N] in [0,1]
        # 只罚超过 margin 的重叠部分
        over = (iou - iou_margin).clamp(min=0.0)            # [N,N]
        # 屏蔽自身对
        eye = torch.eye(N, device=device, dtype=torch.bool)
        over = over.masked_fill(eye, 0.0)
        # 构造 matched 掩码,保护 matched-matched 对
        if protect_matched and matched_src_idx[b].numel() > 0:
            m = torch.zeros(N, dtype=torch.bool, device=device)
            m[matched_src_idx[b]] = True
            mm = m[:, None] & m[None, :]                    # matched-matched
            over = over.masked_fill(mm, 0.0)
        total = total + over.sum()
        n_pairs += N * (N - 1)
    return total / max(n_pairs, 1)


def _pairwise_tiou(spans_xx, eps=1e-6):
    """spans_xx: [N,2] (st,ed) → [N,N] 时序 IoU。"""
    st, ed = spans_xx[:, 0], spans_xx[:, 1]
    inter = (torch.min(ed[:, None], ed[None, :])
             - torch.max(st[:, None], st[None, :])).clamp(min=0)
    len_i = (ed - st).clamp(min=0)
    union = len_i[:, None] + len_i[None, :] - inter
    return inter / union.clamp(min=eps)
````

**四步落地规划(从甜点配置出发,严控变量)**

**第一步——基底选 #2 不选 #3。** 这是最重要的决策。#3 删了 SA 把去重器也删了,你还得靠 diversity loss 从零重建去重能力,事倍功半;**#2 保留了 SA(自带去重)+ QD(已能把激活数顶到 7),只差"把这 7 个 query 推开"这临门一脚**,diversity loss 正好补这一脚。所以阶段三的基底是 **+SA + QD(k=0.25)+ Diversity Loss**,而不是在 #3 上加。把这组记为 **#6**。

**第二步——`div_coef` 做小步扫描,盯住守护指标。** diversity 是把双刃剑:太弱推不动坍缩,太强会把 query 推到乱七八糟的位置砸掉 mAP(就像 #2 新增 query 干扰主目标那样)。建议 `div_coef ∈ {0.1, 0.3, 0.5, 1.0}` 各跑一次,`div_margin` 先锁 0.5。**主看 GT 覆盖率从 0.55 往 1.0 爬、mR+@5 抬头;同时 mAP≥7.0 / Rej-F1≥74 / G-mIoU@1≥50 一项都不能破。** 找那个"覆盖率明显涨、守护不破"的甜点系数。

**第三步——每个 coef 跑完立刻复用你那两把尖刀验证,不靠单一标量下结论。** 用 `plot_query_span.py` 重画 GT≥2 的散布图看 query 是否真的铺开了(而不是被推到空地上瞎站),用 GT 覆盖率标量量化。**特别警惕一种假阳性**:query 被推开了、覆盖率涨了,但推到的是没有 GT 的空白区 → 这会让 mR+@5 涨一点但 mAP/Rej-F1 掉。这就是 margin 要存在的意义,真出现这情况就调大 `div_margin` 让排斥更温和。

**第四步——确认最优 coef 后,补 2~3 个 seed 复测,把 G-mIoU/Rej-F1 那 1~2 点的波动钉死在噪声带内。** 你表里 #2 的 Rej-F1 是 74.73(反而比基线高),#3 的 73.51 略低,这种 1 点级差异本就是单 seed 噪声;最终入论文的数必须是多 seed 均值±std,否则审稿人一眼看穿。
