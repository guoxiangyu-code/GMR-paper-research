# 要注意的说明

基于项目的评测协议，在进行 Error Analysis、模型调优和后处理方案评估时，需严格注意以下评估集的使用规范与分工：

## 1. 验证集 (Val) 的定位与局限
- **设计初衷**：验证集的设计是为了在训练期间进行**定位导向**的轻量级评测，用于 loss 监控和挑选 `best.ckpt`。
- **数据构成**：默认代码流程下，val 集在推理时过滤掉了所有负样本（null-set），**只保留了正样本**。
- **指标限制**：在此集合的默认预测结果上只能计算 mAP、R@k 等纯粹反映时序定位能力的指标。由于缺少负样本，无法计算拒识指标（AUROC、Rej-F1）和端到端的综合指标（G-mIoU）。
- **避坑指南**：**不要直接使用现成的 val 预测结果去分析拒识误报 (FP) 或做端到端指标的参数搜索**。如果一定要在 val 集上做完整的 GMR 诊断，必须强制修改预测生成脚本，开启 `keep_empty_gt=True`，使 null-set 样本进入推理环节，并使用官方工具 `eval/eval_main.py` 进行评测。

## 2. 测试集 (Test) 的完整性
- **设计初衷**：用于评估完整的 GMR 任务表现，包含拒识、时序定位以及端到端性能。
- **数据构成**：全量数据，显式保留了空 GT 样本（负样本）。
- **评测工具**：必须使用官方提供的独立工具 `eval/eval_main.py`。该脚本通过 `normalize_ground_truth(..., drop_empty_gt=False)` 确保正负样本均参与评测，从而能够正确计算出真正的完整指标。

## 3. 对后续 Phase 4 及之后工作的影响
- **阈值搜索与参数调整**：由于需要依赖端到端指标（如 G-mIoU）和拒识指标（如 Rej-F1）来寻找最佳的后处理阈值（例如置信度过滤阈值、Soft-NMS 参数等），我们在进行 parameter sweep 时，**数据基准应当使用 test 集**（如目前 `postprocess.py` 的做法），或者必须手动生成一份包含负样本的完整 val 预测结果。
- **结论与指引**：在后续的代码开发中，务必牢记这一验证集和测试集在“数据分布”和“指标计算”上的严格分工，避免用纯正样本的数据去评估或优化整体系统的拒识能力。
