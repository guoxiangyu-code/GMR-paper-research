# 任务计划：广义时刻检索 — 系统性错误分析与定向改进

## 目标

对 Soccer-GMR 基准上的 Moment-DETR-GMR 基线进行系统性的、数据驱动的错误分析，量化失效模式，通过反事实（oracle）修复对瓶颈进行排序，并实现一个针对最高影响瓶颈的最小化模块。所有结论必须基于官方评估脚本的可重现统计数据。

## 约束条件

1. **不引入大型框架** — 首先定位主要失效模式，然后构建一个小型、有效的模块。
2. **数据驱动** — 所有结论基于官方评估指标（`eval/eval_main.py`，`eval/metrics.py`）。
3. **可重现** — 使用现有代码库、特征和检查点。
4. **FlashVTG-GMR 不可用** — 使用 Moment-DETR-GMR（可用）完成完整流程；注意最终结论最好能在最强基线上验证。

## 基线背景

- **模型**：Moment-DETR-GMR（DETR 风格编码器-解码器 + GMR 适配器存在性预测头）
- **数据集**：Soccer-GMR 标准集划分（训练集：4,138 / 验证集：465 / 测试集：1,036）
- **特征**：CLIP 视觉（512d）+ SlowFast（2304d）+ TEF（2d）= 2818d 视频；CLIP 文本（512d）
- **检查点**：`results/moment_detr_gmr/best.ckpt`（58.7 MB）
- **现有预测**：测试集（1,035 行），验证集（464 行）— 已生成
- **论文阈值**：τ = 0.4（存在性分数门控）

## 阶段 1 — 构建错误分析详情表（`error_analysis.py`）
- **状态**：`未开始`
- **目标**：在不修改官方评估逻辑的前提下，构建逐查询的错误分解表。
- **任务**：
  1. 创建 `eval/error_analysis.py` — 复用 `eval/metrics.py` 中的匹配函数
  2. 按 `qid` 对齐提交与 GT；分配场景标签：`空集`（|G|=0），`单时刻`（|G|=1），`多时刻`（|G|≥2）
  3. 对每个正样本：在 IoU 阈值 {0.5, 0.55, ..., 0.95} 下进行贪心一对一匹配；记录匹配/遗漏/多余计数及每次匹配的 IoU
  4. 对每个空集样本：记录 `pred_exist_score` 以及是否错误接受（分数 > τ）
  5. 输出逐查询详情 CSV/JSONL：`qid`，场景，|G|，|pred|，匹配数，遗漏数，多余数，平均 IoU，pred_exist_score，错误类型标签
- **交付物**：`eval/error_analysis.py`，详情输出文件
- **涉及文件**：`eval/metrics.py`（复用 `greedy_match`，`compute_temporal_iou_batch_cross`）

## 阶段 2 — 量化六种失效模式
- **状态**：`未开始`
- **目标**：量化每种失效模式的比例及典型案例。
- **任务**：
  1. **拒绝假阳性**（空集被接受）：绘制 FP 曲线 vs `pred_exist_score` 阈值；寻找最高分语义相似负样本作为案例研究
  2. **拒绝假阴性**（正样本被错误拒绝）：FN 比例及对 mAP 的影响
  3. **多时刻遗漏**（仅命中首个）：对于 |G|≥2，计算"首个时刻命中率" vs "后续时刻命中率"；量化"仅命中首个"的比例
  4. **多余检测**：|pred| > |G| 的比例；冗余框的分数分布
  5. **边界不精确**：匹配对的 IoU 直方图；"接近命中"（0.3 ≤ IoU < 0.5）的百分比
  6. 汇总表 + 关键分布图；用一句话确定主要瓶颈
- **交付物**：汇总表，分布图（保存到 `results/error_analysis/`），瓶颈识别
- **涉及文件**：阶段 1 的输出，可视化脚本

## 阶段 3 — 用反事实（Oracle）修复对瓶颈排序
- **状态**：`未开始`
- **目标**：通过 oracle 修复带来的指标提升对瓶颈排序，而非凭直觉。
- **任务**：
  1. 修复多时刻遗漏：将遗漏的后续 GT 视为正确召回 → 重新评估
  2. 修复边界：将所有匹配对的 IoU 设为 1.0 → 重新评估
  3. 修复假阳性：正确拒绝所有空集样本 → 重新评估
  4. 修复多余检测：移除所有冗余预测 → 重新评估
  5. 记录每种修复的 G-mIoU@1、mAP、Rej-F1 提升
  6. 按指标提升排序；选择影响最大的方向
  7. 映射到策略：多时刻遗漏 → 覆盖感知检索；FP → 候选验证；边界 → 边界细化
- **交付物**：反事实结果表，排序后的瓶颈列表，所选方向及理由
- **涉及文件**：`eval/eval_main.py`，修改后的预测文件
